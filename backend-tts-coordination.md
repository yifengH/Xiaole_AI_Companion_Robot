# 后端 TTS 下行链路配合排查说明

## 1. 问题现象

设备端当前在播放较短回复时基本正常，但在播放较长回复时，常出现以下问题：

1. 前半段语音正常。
2. 后半段语音出现异常，表现为：
   - 播放节奏明显变快。
   - 多个片段像“挤在一起”。
   - 有时伴随尾部内容丢失或听感混乱。
3. 如果一边播报一边插话，设备端和服务端对“打断”的体验也会变得不稳定。

## 2. 当前设备端结论

对照原版工程后，设备端已经尽量回到接近原版的播放链路：

1. 收到一个下行二进制帧。
2. 将其当作一个 Opus 音频包处理。
3. 解码成 PCM。
4. 直接送到音频输出。

当前设备端仅保留了两项必要兼容修复：

1. 当 `hello` 没有给出合法 `frameDuration` 时，默认回退到 `20ms`。
2. 当 Opus 解码输出缓冲区不够时，按解码器返回的 `needed_size` 扩容后重新解码一次。

这说明：

设备端已经尽量减少“额外调度策略”带来的副作用。  
如果此时长句后半段仍然异常，那么问题更大概率出现在后端下行音频发送策略，而不是字幕流、状态机表层逻辑或者单纯的采样率配置。

## 3. 当前怀疑的核心原因

最需要关注的是后端 TTS 音频的**下发组织方式**，包括但不限于：

1. `hello` 中没有明确给出下行音频的完整参数。
2. 实际发送时没有严格做到“一个二进制帧 = 一个完整 Opus packet”。
3. 后端把一整段 TTS 音频在很短时间内快速灌给设备，而不是按接近实时播放节奏发送。

对当前设备实现来说，第 3 点尤其关键。

设备侧目前没有复杂的播放器时钟调度和成熟的抖动缓冲系统。  
如果服务端把几秒钟的音频在极短时间内全部推送完，设备端就容易在长句后半段出现堆积、丢包、节奏异常或“挤压感”。

## 4. 建议后端配合修改

## 4.1 在 `hello` 中明确下发音频参数

建议后端在 `hello.data.audio` 中明确提供以下字段：

```json
{
  "type": "hello",
  "data": {
    "sessionId": "xxx",
    "audio": {
      "format": "opus",
      "sampleRate": 24000,
      "frameDuration": 20,
      "channels": 1
    }
  }
}
```

建议至少保证：

1. `format = opus`
2. `sampleRate = 24000`
3. `frameDuration = 20`
4. `channels = 1`

## 4.2 严格保证一个二进制帧只包含一个完整 Opus packet

请后端确认：

1. 不要把多个 Opus 包拼在一个二进制帧里发送。
2. 不要把一个 Opus 包拆成多个二进制帧发送。
3. 每个下行 WebSocket 二进制帧都对应一个完整、独立、可直接解码的 Opus packet。

这是当前设备端链路最重要的前提之一。

## 4.3 不要“瞬间灌包”，要按接近实时节奏发送

建议后端对 TTS 下行增加一个 pacing 层，按接近实时的节奏发送：

1. 如果使用 `20ms` Opus packet，则按约 `20ms/packet` 下发。
2. 允许有少量网络抖动，但不要成批突发发送。
3. 不要在 `tts.start` 之后立刻把整段 TTS 全部快速推完。

不建议的方式：

1. `tts.start`
2. 几秒钟的全部音频包在几十毫秒到几百毫秒内全部下发
3. `tts.stop`

建议的方式：

1. `tts.start`
2. 每约 `20ms` 下发 1 个 Opus packet
3. 全部发送完成后下发 `tts.stop`

## 4.4 如有条件，增加下行包序号或时间戳

建议后端在下行音频链路中增加调试信息，便于定位：

1. 每个下行音频包增加 `seq`。
2. 或者增加发送时间戳。
3. 至少在服务端调试日志中可以看到：
   - 包序号
   - 生成时间
   - 发送时间
   - 是否批量发送

这样可以快速判断是否存在：

1. 包突发发送
2. 包乱序
3. 包重复
4. 包间隔不稳定

## 5. 建议后端优先检查的内容

建议后端按以下顺序排查：

1. `hello` 是否明确下发了 `sampleRate=24000`、`frameDuration=20`、`channels=1`。
2. 每个 WebSocket 二进制帧是否严格只承载一个完整 Opus packet。
3. 下行 TTS 是否存在“整段音频快速推送”的行为。
4. 下行发送间隔是否接近实时节奏。
5. 是否存在某些长句在服务端侧被重新分块、聚合或压缩发送。

## 6. 设备端建议的联调验证方法

后端修改后，建议用下面的方法验证：

1. 使用较短回复测试：
   - 确认基本播放正常。
2. 使用较长回复测试：
   - 如“讲一个长一点的故事”。
   - 重点观察后半段是否仍然出现变快、挤压、尾部异常。
3. 使用打断测试：
   - 在设备播报过程中主动插话。
   - 确认设备是否能及时停止当前播报并响应新的输入。
4. 如果后端能输出发送日志：
   - 记录每个下行包的发送时间差。
   - 重点看是否真的接近 `20ms/packet`。

## 7. 可以直接转给后端同事的简版结论

当前设备端已经尽量回归原版播放链路，长句后半段异常问题仍然存在，现怀疑根因在后端 TTS 下行发送策略。请后端重点配合以下几点：

1. 在 `hello.data.audio` 中明确下发 `format=opus`、`sampleRate=24000`、`frameDuration=20`、`channels=1`。
2. 严格保证一个 WebSocket 二进制帧只包含一个完整 Opus packet。
3. 不要把整段 TTS 音频一次性快速灌给设备，请按接近实时的节奏发送，建议 `20ms/packet`。
4. 如有条件，请给下行音频增加包序号或时间戳，便于排查是否存在突发、乱序、重复或异常分包。
